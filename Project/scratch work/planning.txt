In our working example, we must display each state in its full unsimplified form ... so no nesting
We should have more flexibility when working with syntax

We will need two types of vectors: 
    

(state W (superposition (state W) (state B))) = W{W, B} ?


(superposition (state W B) (state W W))
    - states must all have the same number of qubits in them, as each represents an entire possible state 
        of the system
    - this example is {WB, WW} or W{B, W}
    - but can we represent this directly as W{B, W}?

(state W (superposition W B))
    - once we "move down" a level into a superposition when it's nested like this, we are only working with that
        particular superposition
    - ex: in W{W, B}, our superposition is only concerned with the latter qubits
    - how about states with more qubits?

W{WB, BW} ==> either WWB or WBW
(state W (superposition (state W B) (state B W)))
    - if we operate like this, we need to adjust our target
    - H1(W{WB, BW}) ==> W{WB, BB, WW, bW} ==> W{W{W, B}, {W, b}B}
    - order:
        - (hadamard (state W (superposition (state W B) (state B W))) 1)
        - (hadamard (superposition (state W B) (state B W)) 0)   ;sub 1 from target, b/c first qubit
            does not need to be operated on
        - (hadamard (state W B) 0) (hadamard (state B W) 0)   ; separate into two instances of hadamard 
            operation for each superposition, return resultant states
        - (superposition (state (superposition W B) B) (state (superposition W b) W))  ; now, we have two outcomes 
            from each of our hadamards, so we superimpose them
        - (state W (superposition (state (superposition W B) B) (state (superposition W b) W)))  ; finally, we insert 
            the superposition back where it belongs in the original state
            this directly translates to W {{W, B} B, {W, b} W}

Grammar
state ==> (state x ...) where x is:
    - W
    - B
    - w
    - b
    - (superposition y ...)

superposition ==> (superposition y ...) where y is:
    - W 
    - B 
    - w 
    - b 
    - (state x ...)

So:
    - states can contain superpositions, representing possible substates 
    - superpositions contain states, each of which are a possible outcome 
    - this system needs some more testing!

3/29

Now that the core is built out, I think it's time we discuss how circuits will be modelled.

(operate <in-state> <circ>)

(circ wire ...)
    - we will pass in a list of wires that represent each wire in the circuits

(wire gate ...)
    - and each wire will consist of a series of gates like this
    - these gates will not be our actual gate functions (as those need states to function), they'll instead be 
        s-exps which will be parsed to generate the actual gate functions   

ex of expansion:
    - (operate WW (circ (wire 0 identity CNOT:T) (wire 1 hadamard CNOT:C)))
        ==>
    - (CNOT (hadamard (identity WW) 1) 1 0)

How will this work? More in detail
    - grab state from operate function
    - find first gate on each wire
        - in this case, identity and hadamard 
        - apply in arbitrary order, as these are independent gates
        - so: (hadamard (identity WW) 1)
    - now, we have CNOT:T and CNOT:C on both
        - we can grab wire numbers from given wires
        - whatever processing function we use will recieve input in the form (circ (wire 0 CNOT:T) (wire 1 CNOT:C))
            - scan with catch function (circ wire ...)
        - we see we have CNOT:T and CNOT:C,


Ok I'm realizing this system isn't gonna work so let's try something else 

A more iterative structure:
    - (operate WW (circ (hadamard 0) (CNOT 0 1)))
    - catch circuits with: (operate state (circ gates ...))
        - for each gate:
            - generate (<gate> state)
            - ex for CNOT: 
                - (CNOT 0 1) ==> (CNOT state 0 1)
            - ex for H => CNOT as above:
                - (circ (hadamard 0) (CNOT 0 1)) ==> (CNOT (hadamard state 0) 0 1)
            - after each gate evaluation in the for loop, update state to be (<gate> state)
                - state = WW, (circ (hadamard 0) (CNOT 0 1)) --> 
                - state = (hadamard WW 0), (circ (CNOT 0 1)) -->
                - state = (CNOT (hadamard WW 0) 0 1), return evaluation!

Bottom line - cause of how circuits work, it's much better to have more of a linear/iterative approach
to circuits, to generate a recursive state evaluation
